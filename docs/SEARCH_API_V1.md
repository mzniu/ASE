# AI 联网搜索服务 — API 与产品说明（v1）

面向 **开发者** 与 **Agent 工具链**：无状态、单端点，用用户级 API Key 认证，返回 **Markdown 正文**（非固定字段，由服务端按检索结果组织，不采用「给人看的搜索结果列表 UI」形态）。

---

## 1. 产品定位

| 项目 | 说明 |
|------|------|
| 使用者 | 开发者在 Agent / AI 工具里配置「联网搜索」时集成本服务 |
| 价值 | 为 Agent 提供 **事实与工具检索** 导向的内容；正文经整理，便于直接作为模型上下文 |
| 非目标 | 不提供面向终端用户的搜索门户体验；初期不提供 SLA；不提供高并发爬站能力 |

---

## 2. 功能与非功能需求（已确认）

### 2.1 数据来源与策略

1. **公开网页**：通过第三方搜索能力（例如 Baidu API、Bing API、Tavily API 等）获取候选与内容，并结合 **自建索引**。
2. **命中「足够」时**：优先从自建索引直接生成响应，降低对外部 API 与抓取的依赖。
3. **「足够」的判定**（需同时可配置/可调参，具体阈值实现期确定）：  
   - 命中条数  
   - 文本长度  
   - 相似度分数  
4. **索引不足时**：**回落**到实时搜索/抓取路径，再整理为 Markdown 返回。

### 2.2 实时性与体验

- Agent 场景下 **实时性可略差**，但内容须 **有用且经整理**。
- **仅支持同步**：成功时 **`200 OK`**，响应体为 **一段 Markdown**（见下文）。
- 因存在「回落到实时搜索/抓取」，服务端须定义 **单请求处理上限** 与 **客户端建议超时**（实现期写入配置；本文档预留「运维参数」一节）。

### 2.3 合规与运营

- **不允许**将本服务用作高并发、大规模对外爬站；对外说明与限流策略需与此一致。
- **数据驻留（已确认范围）**：**自建存储与自建索引**必须部署在指定境内区域；查询经境外第三方搜索 API（如 Bing、Tavily 等）流转时，可能涉及 **数据出境或境外处理**，须在《隐私政策》/《数据处理说明》中向用户披露，并与法务评审结论一致。
- 依赖第三方搜索 API 时，须在隐私与数据处理说明中列出 **可能的数据出境/第三方处理**（与驻留策略一并评审）。

### 2.4 鉴权与后续能力

- **用户级 API Key**：前期仅用于 **认证**；计费、审计等 **后期再扩展**。

## 3. API 契约（MVP）

### 3.1 端点

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/v1/search` | 主业务端点；请求体见 §3.3 |
| `POST` | `/v1/documents` | **可选**：向自建索引写入文档（`id`、`title`、`body_text`）；需配置 OpenSearch；未配置时 **501**；需 Bearer 鉴权，与 `/v1/search` 同属限流组 |
| `GET` | `/health` | **探活**（运维/编排器使用）：**无鉴权**、**不限流**；`200` + `Content-Type: application/json`，body 为 `{"status":"ok"}` |
| `GET` | `/metrics` | **Prometheus** 指标（如 `ase_search_orchestration_total`）；**无鉴权**、**不限流**（与 `/health` 一致，供抓取器使用） |

用量查询等端点若需要可在网关或内网扩展。

### 3.2 认证

- 请求头：`Authorization: Bearer <api_key>`
- `<api_key>` 为用户级密钥；无效或缺失时返回 **401**（见错误约定）。

### 3.3 请求

- **Header**  
  - `Content-Type: application/json`  
  - `Authorization: Bearer <api_key>`
- **Body（JSON）**

```json
{
  "query": "string",
  "providers": ["baidu", "bing", "google"]
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `query` | string | 是 | 自然语言查询；长度上限实现期定义（建议文档化默认如 4k 字符，超限 **400**） |
| `providers` | string[] | 否 | 选用哪些搜索引擎；合法值为服务端已注册的名称（小写）：`baidu`、`bing`、`google`、`tavily`、`stub` 等。**省略**或空数组时使用环境变量 **`SEARCH_DEFAULT_PROVIDERS`**；若未配置则按 **baidu → bing → google → tavily** 顺序取第一个已启用的引擎。可填多个，结果按 URL 去重合并，并标注 **来源**。未知名称返回 **400**。 |

本版本 **不** 在请求体中暴露 `locale`、`time_range`、`response_format` 等参数；若后续需要，可在 **v2** 或同路径下扩展字段并保持向后兼容。

#### 3.3.1 `POST /v1/documents`（可选，索引写入）

- **Header**：与 §3.2 相同（`Content-Type: application/json`，`Authorization: Bearer`）。
- **Body（JSON）**：`id`（必填，文档 `_id`）、`title`、`body_text`（与索引映射字段一致）。
- **成功**：**204 No Content**（无 body）。
- **未配置 OpenSearch**：**501**，Problem Details（见 §3.5）。

### 3.4 成功响应

- **HTTP**：`200 OK`
- **Content-Type**：`text/markdown; charset=utf-8`
- **Body**：**一整段 Markdown**，语义上为「可直接并入模型上下文的整理结果」。
  - **非固定字段**：不强制固定 JSON schema；结构由服务端根据检索与整理策略变化。
  - **呈现约定**：**不**采用传统搜索引擎「十条蓝色链接 + 摘要」的列表 UI 形态；若需可追溯性，可在 Markdown 内以 **非列表门户式** 的方式呈现（例如文末参考、脚注式简短引用），具体样式由产品/实现统一，并在《响应风格指南》中细化（可另文）。
  - **长度与截断（已确认）**：当单次处理接近 **服务端时限或输出预算** 时，服务端 **硬截断** 已生成内容，**仍返回 `200 OK`**，响应体为 **部分 Markdown**；建议在正文末尾或固定位置用简短说明标示「内容已截断」（具体措辞由《响应风格指南》统一）。**不因截断而改用 202 或错误码**，除非完全无法产出任何可用片段（见 §4.2）。

### 3.5 错误响应

成功路径只有 Markdown；**失败路径**建议对开发者友好、可机读：

- **Content-Type**：`application/problem+json`（RFC 9457 Problem Details）或项目统一的 JSON 错误体（二选一并全文档统一）。
- 至少区分：**401**（鉴权）、**400**（参数非法）、**429**（限流）、**504**（例如在时限内 **无法产生任何可用 Markdown 片段** 或网关上游超时）、**503**（依赖不可用）、**500**（内部错误）。
- **与 §3.4 的区分**：已能产出 **部分** 可用 Markdown 时，**不** 使用 **504**，应 **200 + 截断后的 Markdown**。

（Problem Details 的 `type` / `title` / `detail` 字段表实现期补全。）

## 4. 内部行为说明（供实现与评审，非对外保证）

以下内容用于对齐开发，**不构成 SLA**。

### 4.1 索引优先与回落

```
请求 query
    → 评估自建索引是否「足够」（条数、长度、相似度等阈值）
        → 足够：基于索引生成 Markdown → 200
        → 不足：调用外部搜索 API / 受控抓取 → 整理 → 可异步写入索引（实现细节）→ 200
```

### 4.2 同步与超时

- 对外仅 **同步 200 + Markdown**；须在服务端设置 **硬超时** 与 **输出长度/Token 预算**。
- **截断优先**：在触及时限或预算前若已生成部分内容，则 **返回 200 + 部分 Markdown**（见 §3.4），**不** 将「可返回部分结果」的情况判为 **504**。
- **完全失败**：在时限内 **无法形成任何可用片段**（或关键依赖全部失败且无降级）时，返回 **504/503** + Problem Details，**不** 使用 202 轮询。
- 建议在文档「集成建议」中写明：**客户端超时建议 ≥ 服务端上限 + 网络余量**；客户端应能消费 **可能被截断** 的 Markdown。

### 4.3 限流与滥用防护

- 针对每 API Key 与全局限流，防止被用作高并发爬站；具体 QPS/每日配额 **实现期配置**。
