# 测试策略与 TDD 规范

本项目的默认开发方式是 **TDD（测试驱动开发）**：先写 **失败测试**（红）→ 最小实现（绿）→ 重构；与 [SRS.md](./SRS.md) 中的 `REQ-*` 及 [SEARCH_API_V1.md](./SEARCH_API_V1.md) 契约对齐。

---

## 1. 原则

1. **可重复**：测试不依赖手动步骤、不依赖真实外网（默认使用 **测试双倍**：Fake/Stub/Mock）。
2. **快速**：单元测试秒级反馈；慢速集成测试在 CI 中分层执行（见 §6）。
3. **可读**：测试名称说明 **行为**（Given-When-Then 或等价命名），而非实现细节。
4. **确定性**：避免与时间、随机、全局可变单测耦合；必要时注入时钟或随机种子。

---

## 2. TDD 工作流（强制于功能代码）

对 **新增或修改业务行为** 的提交，推荐按以下循环：

1. **红**：编写 **一条** 失败测试，表达期望行为（对应 `REQ-*` 或 API 契约条款）。
2. **绿**：写 **最少** 代码使测试通过。
3. **重构**：在不改变行为的前提下整理代码；测试保持绿色。
4. 重复直至该行为覆盖完整；再处理边界与错误路径。

**例外（须在 PR 说明理由）**：纯配置、纯文档、纯重构且无行为变化——仍建议跑全量测试；紧急热修可事后补测试但须登记技术债。

---

## 3. 测试分层

| 层级 | 目的 | 典型内容 | 依赖 |
|------|------|----------|------|
| **单元测试** | 纯逻辑、无 I/O 或内存双倍 | 「足够」判定、Markdown 拼装、截断 | 无网络、无真实索引 |
| **契约测试** | HTTP 对外行为与文档一致 | 方法、路径、状态码、`Content-Type`、401/400 | 内存 HTTP 服务器或框架测试客户端 |
| **集成测试** | 组件协作：路由 + 假索引 + 假 Provider | 索引命中 vs 回落路径 | 容器内嵌、临时目录或 Testcontainers（选型后定） |

**禁止**：在默认 CI 中调用 **真实** Baidu/Bing/Tavily 密钥或生产索引。

---

## 4. 测试命名与组织（语言无关约定）

- 目录建议（实现落地后二选一或组合）：
  - `test/`、`tests/` 或与源码同级的 `*_test.go` / `test_*.py` —— 与所选语言惯例一致。
- 命名示例：
  - `TestSearch_Returns200Markdown_WhenQueryValid`
  - `TestSearch_Returns401_WhenBearerMissing`
- **每个** 与需求相关的测试文件顶部或类注释中可标注：`Covers: REQ-F-003, REQ-F-004`。

---

## 5. 覆盖率

- **目标**：核心业务逻辑（判定、整理、截断）覆盖率 **高**；具体阈值在首次实现稳定后写入 `scripts/run-tests.sh` 或语言专属配置（如 `go test -coverprofile` / `pytest --cov`）。
- **原则**：覆盖率是 **信号** 而非目的；禁止为数字而测无断言的「空测试」。

---

## 6. CI 中的执行策略

- **每次 PR / push 主分支**：至少运行 **单元测试 + 契约测试**；`scripts/run-tests.sh` 会探测项目类型并执行（见仓库根脚本）。
- **集成测试**：若慢或需 Docker，可设为 **工作流单独 job** 或在合并前必跑（在 [GITHUB_WORKFLOW.md](./GITHUB_WORKFLOW.md) 中锁定）。
- **文档校验**：`scripts/verify-docs.sh` 确保立项/SRS/TDD 等关键文档未被误删。

---

## 7. 与 SRS 的追踪

- 新功能：在 PR 描述中列出 **REQ-* ID**。
- 缺陷修复：增加 **回归测试**，并在 PR 中引用对应 `REQ-*` 或 Issue。

---

## 8. 测试数据与夹具

- 使用 **匿名化**、**可公开** 的短文本作为样例 query 与索引片段。
- 大型夹具放 `testdata/` 或 `fixtures/`，避免塞在源码字符串中难以维护。
