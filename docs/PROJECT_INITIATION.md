## 1. 背景与问题陈述

- **背景**：大模型与 Agent 需要「可接入的联网检索」能力，传统面向人类的搜索 API（链接列表、snippet）与 Agent 所需的「可直接并入上下文的整理文本」不匹配。
- **要解决的问题**：为开发者提供 **单一、无状态、Key 鉴权** 的搜索能力，输出 **Markdown**，便于在 Agent 工具链中配置「联网搜索」。
- **不做的事**：不提供面向终端用户的搜索门户；初期不提供 SLA；不提供高并发对外爬站能力。

---

## 2. 目标与成功标准

| 维度 | 目标 | 可验证信号（MVP） |
|------|------|-------------------|
| 集成 | 开发者可用 HTTP 客户端在数分钟内完成首次成功调用 | 文档含可复制示例；CI 覆盖契约测试 |
| 输出形态 | 响应为 **Markdown**，非「十条链接」式 UI 列表 | SRS 与 API 文档一致；契约/冒烟测试断言 Content-Type 与语义 |
| 数据策略 | 索引优先，不足则回落实时检索/受控获取 | 架构文档与集成测试（测试双倍）可演示两条路径 |
| 合规 | 存储与索引 **境内**；第三方跨境处理可披露 | 架构与隐私说明一致；部署文档标明区域 |
| 工程 | **TDD** 为主力开发方式；主分支受 CI 保护 | [TESTING_AND_TDD.md](./TESTING_AND_TDD.md) 与 GitHub Actions 生效 |

## 3. 范围（MVP）

### 3.1 在内

- 对外 **仅** `POST /v1/search`；请求体 **仅** `query`（JSON）。
- 认证：**用户级 API Key**，`Authorization: Bearer`。
- 成功响应：**200**，`text/markdown; charset=utf-8`；正文结构 **非固定字段**，由服务整理。
- 索引「足够」判定：命中条数、文本长度、相似度（阈值可配置）；不足则 **回落** 第三方搜索 API / 受控抓取。
- 时限/预算下 **硬截断**：仍 **200 + 部分 Markdown**；完全无法产出可用片段时 **504/503** 等（见 API 文档）。
- 限流与滥用防护（防爬）。

### 3.2 不在内（本阶段）

- 计费、审计、多租户控制台（可后续迭代）。
- 对外 `/health`、索引管理 API（运维可用旁路探测，不纳入对外契约）。
- 对外 SLA 承诺。

---

## 4. 假设与约束

- **技术**：代码托管 **GitHub**；合并须通过 **CI**；功能开发以 **TDD** 为默认流程。
- **合规**：自建 **存储与索引** 部署在 **境内**；经境外搜索 API 的数据流须在隐私/数据处理说明中披露。
- **商业**：依赖 Baidu、Bing、Tavily 等 **第三方 API**，受各供应商条款与可用性约束。

## 6. 里程碑（建议）

| 阶段 | 交付物 | 完成判据 |
|------|--------|----------|
| M0 | 文档体系 + CI（文档校验 + 可扩展测试任务） | 主分支合并即跑通 CI |
| M1 | 最小可运行服务：`/v1/search` 返回固定/桩 Markdown + 鉴权 | 契约测试绿；无密钥泄漏 |
| M2 | 索引路径 + 「足够」判定 + 回落桩 | 集成测试覆盖两条路径 |
| M3 | 接真实第三方 API（可配置）+ 受控抓取策略 | 配置化密钥；限流生效 |

（日期由项目负责人填入项目计划。）

## 7. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| 第三方搜索 API 变更或限流 | 可用性下降 | 抽象 Provider 接口；集成测试使用双倍；文档不承诺超出供应商能力 |
| 合规解读变化 | 部署或披露需调整 | 架构与隐私文档版本化；重大变更走评审 |
| TDD 执行不到位 | 回归成本上升 | PR 模板强制测试说明；CI 失败不可合并 |
